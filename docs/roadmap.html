Phase 2 — Sân khấu 3D & Hero Section / 3D Canvas & Hero Overlay

Hệ thống phân tầng đồ họa (Tier), Canvas, Hạc (Crane), Lụa (Silk), và Text overlay.

⚠️ High Risk

Mô tả Kiến trúc 3D Tier / 3D Adaptive Tiers

Để tránh giật lag trên máy yếu và mobile, chúng ta chia làm 3 Tier chính dựa vào detect-gpu và prefers-reduced-motion:

  • Tier A (Cinematic - Desktop mạnh): GLB Hạc đầy đủ chi tiết + Bloom postprocessing + 2000+ hạt tinh vân chuyển động + Motion Tilt.
  • Tier B (Lite - Mobile/GPU yếu): GLB Hạc dạng Low-Poly Draco, tắt Bloom, giảm hạt bụi xuống ≤600 hạt, tắt chuyển động phụ.
  • Tier C (Still - a11y/Yếu hẳn): Không load WebGL/Canvas. Sử dụng ảnh tĩnh dạng 2.5D Layered-PNG xếp đè, dịch chuyển nhẹ bằng CSS/motion.

Giảm thiểu rủi ro Hydration / Hydration Mitigation

  • Bắt buộc import dynamic các file chứa Canvas R3F ở client: const Scene = dynamic(() => import('./Scene'), { ssr: false }).
  • Hiển thị HeroPoster (ảnh WebP render sẵn của Hero) làm placeholder ngay lập tức để đạt LCP nhanh (< 2.5s), Canvas chỉ tải ngầm sau đó.
TASK 2.1 src/three/tier.ts

Viết logic nhận dạng thiết bị và chọn Tier đồ họa tier.ts.

Đọc thông số từ detect-gpu, lắng nghe CSS query prefers-reduced-motion để ép về Tier C.

TASK 2.2 Scene.tsx

Dựng R3F Canvas Container tại src/three/Scene.tsx.

Tích hợp PerformanceMonitor, AdaptiveDpr, và ScrollControls để xoay camera mượt dựa trên thanh cuộn thực tế.

TASK 2.3 Placeholders 3D

Tạo các cấu phần 3D placeholder (dùng primitive hình hộp/mặt phẳng trước khi có asset thật).

  • Crane.tsx: placeholder hạc (mesh hình hộp).
  • Silk.tsx: lụa đỏ chuyển động uốn sóng sine nhẹ.
  • Starfield.tsx: hệ thống hạt bụi sao lung linh.
  • Bloom effect nhẹ ở hậu kỳ (Postprocessing).
TASK 2.4 Tier C Fallback

Dựng component LayeredPNG.tsx cho Tier C.

Dùng framer-motion tạo hiệu ứng parallax 2.5D giữa các lớp ảnh PNG chồng nhau khi người dùng rê chuột.

Phase 5 — Form Liên Hệ & API Gửi Mail / Contact API Route & Form

Form UI, validation Zod, API Route, Honeypot chống spam, tích hợp Resend SDK.

P5-API

An toàn & Chống Spam / Form Security & Anti-spam

  • Ẩn Email: Tuyệt đối không render email dạng text ở frontend để tránh bot quét.
  • Honeypot: Tạo một trường input ẩn company bằng CSS. Nếu bot điền vào, route server sẽ âm thầm từ chối (trả về status 200 thành công giả để bot tưởng đã spam được).
  • Zod Validation: Validate chặt chẽ dữ liệu đầu vào cả ở client và server.
  • Rate-limiting: Khống chế tối đa 5 mail/giờ trên một IP (lưu cache IP đơn giản trên server bằng Map tạm thời/Redis).
TASK 5.1 ContactForm UI

Dựng UI Form liên hệ ở chân trang chủ (Footer / Section).

Input nhập Tên, Email, Lời nhắn. Báo lỗi đỏ rõ ràng dưới mỗi input nếu nhập thiếu hoặc sai định dạng.

TASK 5.2 api/contact/route.ts

Viết Route API xử lý gửi thư tại src/app/api/contact/route.ts.

Tích hợp Zod parser, Honeypot (check body.company), check rate-limit theo IP, và dùng Resend SDK để chuyển tiếp email về mail cá nhân của bạn.

TASK 5.3 Dev Logging fallback

Cấu hình ghi log fallback ở môi trường local.

Nếu ở môi trường dev mà thiếu RESEND_API_KEY, route sẽ in thẳng nội dung ra console thay vì quăng lỗi, tránh chặn đứng tiến độ chạy thử ở local.

Kiểm thử / Validate: Nhập form thử, server báo nhận OK, ở môi trường dev in log chuẩn, spam quá 5 lần báo block.
Độ khó: MEDIUM

P6-Optimization

Ngân sách Hiệu năng / Performance Budget (§7.5)

  • LCP < 2.5 giây (Tải poster tĩnh cực nhanh trước, Canvas lazy sau).
  • Chữ chạy ban đầu (initial JS bundle) < 250KB gzipped.
  • Mô hình 3D Hạc crane.glb < 1.5MB sau khi nén Draco.
